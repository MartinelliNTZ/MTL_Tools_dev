MTL Tools - VectorField Plugin Update Instructions
===================================================

This document describes how to bring the **VectorField** plugin into the same
structure and conventions currently used by the rest of the codebase (as of
2026‑03‑03). Follow these steps whenever you need to update or rebase the
VectorField plugin.

1.  **Preferences & global settings**
    - Read the global settings (`MTL Tools Settings`) on plugin init.  The
      same keys used by other tools should be available (`async_threshold_bytes`,
      `async_threshold_features`, etc.).  Use `self.init(..., load_settings_prefs=True)`
      in the constructor.
    - Add local tool preferences using `load_tool_prefs`/`save_tool_prefs` and
      persist any default values (e.g. last buffer distance, last output path).

2.  **User interface**
    - Build UI using `WidgetFactory` helpers (layer input, spinboxes,
      file selectors, QML selector, bottom action buttons).  See
      `GenerateTrailPlugin._build_ui()` for a reference layout.
    - Log each component addition with `self.logger.debug` for easier debugging.
    - Use `self.layout.add_items([...])` and call `self.setLayout(self.layout)`.

3.  **Layer resolution & validation**
    - Factor out `_resolve_input_layer()` which accepts either a QgsVectorLayer or
      layer ID and converts distance units via
      `VectorLayerProjection.convert_distance_to_layer_units()` if needed.
    - Validate the layer geometry using `VectorLayerSource.validate_layer()` and
      display a `QgisMessageUtil.modal_error()` on failure.
    - Implement `_process_selection()` to handle "only selected" logic similarly
      to GenerateTrail.

4.  **Synchronous vs asynchronous logic**
    - Determine feature count with `layer.featureCount()` and compare against the
      threshold from settings.  If larger, invoke `_run_async_pipeline()`;
      otherwise call `_run_sync_pipeline()`.
    - `_run_async_pipeline()` should create an `AsyncPipelineEngine` with
      `ExplodeStep`, `BufferStep`, `SaveVectorStep` (or your own sequence),
      passing in an `ExecutionContext` pre‑populated with parameters.
    - `_run_sync_pipeline()` mirrors the engine steps but runs them directly
      using `VectorLayerGeometry` helpers and captures any exceptions to show
      `QgisMessageUtil.modal_error()`.

5.  **Pipeline feedback & task progress**
    - All tasks used by the engine must subclass `BaseTask` and call
      `self.setProgress()`, `self.pushInfo()`, and honour cancellation via
      `if self.isCanceled():`.
    - Use `TaskFeedback(self)` when invoking processing algorithms or manual
      loops; call `feedback.setProgress(pct)` and check `feedback.isCanceled()`
      periodically.
    - In tasks with early work, nudge progress to 1% immediately so the global
      bar moves.

6.  **Explode/Buffer/Save helpers**
    - Copy the current implementations of `VectorLayerGeometry.explode_lines_to_path_safe`,
      `VectorLayerGeometry.create_buffer_geometry`, and how `VectorLayerSource.save_vector_layer`
      is used above.  They handle file deletion, attribute cleaning (`fid`),
      and generate feedback.

7.  **Logging**
    - Use `self.logger.info/debug/warning/error` liberally in each method to
      track plugin flow.
    - Task classes should call `LogUtils.log(...)` as shown in
      `ExplodeHugeLayerTask`.

8.  **Post‑processing**
    - In `_on_pipeline_finished()` re‑add the final layer to the project if
      needed and apply QML style if the user asked for it.
    - Report completion with `QgisMessageUtil.bar_success()` and call
      `self.finish_stats()`/`self._save_prefs()`.

9.  **Testing & fallback paths**
    - Make sure tasks fall back to native processing (`VectorLayerGeometry.explode_lines_to_path`)
      if the manual version returns false.
    - Catch and log any exceptions in synchronous pipeline code to avoid
      crashing QGIS.

10. **Versioning and rollback**
    - When reverting changes, reverse edits in plugin code and restore
      previous entries in docs or preferences.
    - Keep a copy of this instructions file with the repo so future updates
      can reapply the current format.

---

This text should be executed next time you run a script or used by a colleague
as a checklist. Feel free to copy/paste sections into commit messages or
PR descriptions when porting VectorField to the new architecture.
